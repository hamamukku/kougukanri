# Day12 エラー表示方針（共通）

## 1) 方針（共通）
- 画面ごとではなく、共通の UI 挙動を採用する。
- 表示文言は画面上部赤字で `error: xxx` を基本とし、既存画面実装の上書き方針に合わせる。
- 再送可能性を優先し、可能な場合は入力値・選択状態を保持する。

## 2) middleware 根拠（auth/role）
- `middleware.ts` の `matcher` は `/api` を除外している。
- そのため API からの 401/403 は原則返りにくく、画面遷移ガードとして扱う。
- 非ログイン時: `/login` へリダイレクト（`next` クエリ付き）
- `/admin` アクセス時に role=admin 以外: `/tools` へリダイレクト

## 3) ステータス別運用

### 401
- 挙動: `window.location.href = "/login"`
- 保持: 入力/選択を保全

### 403
- 挙動: `/tools` へ遷移（admin画面では画面内エラーメッセージ表示でも可）
- 根拠: middleware の role ガード

### 409（業務エラー）
- 表示: `error: xxx`
- 位置: 画面上部（画面仕様上必要なら該当行近傍）
- 状態: 入力/選択を保持し、修正後に再試行可能

### 422（入力エラー）
- 表示: 該当入力に近い場所（可能な画面では入力項目付近）
- 補助表示: 併せて画面上部 `error: xxx`
- 状態: 入力値を保持

### 予約API補足
- `POST /api/reservations/confirm` のエラーは 422 を入力系、409 を業務エラーとして扱い、`day12_error_policy` の 422/409 に準拠する。

### 400（暫定）
- 表示: `error: xxx`（画面上部）
- 処理方針: 再実行は可。状態は維持

### 404（暫定）
- 表示: `error: xxx`（画面上部）
- 処理方針: 再取得後に再試行

### 0 / 通信エラー / 500系（暫定）
- 表示: `error: 通信に失敗しました` 系の画面上部赤字
- 再試行: ユーザー操作で再送可能
